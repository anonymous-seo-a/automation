# 設計思想

最終更新: 2026年3月29日

---

## なぜこのシステムを作るのか

soicoの事業を月3000万の売上に乗せ、リタイア資金を確保するまでの期間、
Daikiの認知リソースを最大限に温存しながら事業を回す仕組みが必要。

このシステムの本質は「自動化ツール」ではない。
**Daikiの認知負荷を削減する代理実行基盤**である。

LINEで方針だけ伝えれば、調査・実行・テスト・報告までAIが自律的に行う。
Daikiは判断と承認だけに集中できる状態を作る。

---

## 設計の3原則

### 原則1: 判断と実行を分離する

Daikiが行うのは「何をやるか」の判断だけ。
「どうやるか」「実際に手を動かす」はシステムが担う。

この分離がなぜ重要か：
- Daikiの思考パイプラインは「直感→構造把握→言語化」の順で動く
- 実行の細部に引きずられると、構造把握に使うべきリソースが消耗する
- 「全工程を自分で掌握しようとする」パターン（ナレッジファイル2参照）の
  健全な代替手段として、システムに実行を委任する

### 原則2: 安全側に倒す

- 本番環境への直接変更は行わない（提案→承認→実行の3段階）
- API予算に日次・月次の上限を設ける。超過したら全停止
- 5回リトライして失敗したら人間に判断を返す
- 不明確な指示は実行せず、確認を返す

「暴走して取り返しのつかないことをする」リスクを構造的に排除する。
これはDaikiの「安全感の欠如」という根本課題に対する、
システム設計レベルでの配慮でもある。

### 原則3: 段階的に育てる

一気に全部作らない。

1. まず母艦（LINE連携 + タスク管理）を動かす
2. soicoエージェントを1つだけ繋ぐ
3. 実際に使いながらプロンプトを改善する
4. 安定したら次のエージェントを追加する

これはDaikiの「構造を把握してから動く」パターンと整合する。
全体像が見えないまま大量のコードを書くのではなく、
動く小さな単位を積み上げる。

## アーキテクチャ判断の根拠

### なぜSQLiteか（PostgreSQLやMongoDBではなく）

- VPS 1台で完結する。外部DBサービス不要
- better-sqlite3は同期APIで、Node.jsでの扱いがシンプル
- タスクキューとナレッジDBの規模はSQLiteで十分（月間数千〜数万レコード）
- バックアップはファイルコピーだけで済む
- 将来的にスケールが必要になった時点でPostgreSQLに移行可能
  （SQLの構文はほぼ互換）

### なぜClaude SDKを使わずfetch直接か

- SDKのバージョンアップに依存しない
- レスポンスの生データにアクセスしやすい（usage等のメタデータ）
- デバッグ時にリクエスト/レスポンスの全容が見える
- SDKの型定義とAPIの実際のレスポンスのズレに悩まされない

### なぜExpress（Fastifyではなく）

- LINE Bot SDKの公式ミドルウェアがExpress前提
- エコシステムが成熟しており、Claude Codeが書くコードの精度が高い
- このシステムでHTTPパフォーマンスがボトルネックになることはない

### なぜPM2（systemdではなく）

- `pm2 logs` でリアルタイムログ確認が容易
- `pm2 restart` でゼロダウンタイム再起動
- メモリ上限超過時の自動再起動
- Claude Codeが生成するデプロイスクリプトとの相性が良い

## モデル選択の判断基準

### Sonnet 4.6をデフォルトにした理由

| 観点 | Sonnet 4.6 | Opus 4.6 |
|------|-----------|----------|
| 入力 | $3/1M tokens | $5/1M tokens |
| 出力 | $15/1M tokens | $25/1M tokens |
| 速度 | 高速 | 中速 |
| 精度 | 実用十分 | 最高 |

- タスク解釈、コード生成、定型分析はSonnetで精度が十分
- コスト差は入力で40%、出力で40%
- Sonnet 90% + Opus 10%の構成で月2〜3万円に収まる

### Opusにルーティングする条件

1. `requires_opus: true` がタスク解釈エンジンによって付与された場合
2. リトライが3回目以降（Sonnetで解決できない複雑な問題）
3. SEO戦略判断やデータの総合分析（YMYL領域の正確性が求められる場合）

### 予算管理のロジック

- 日次上限: $1.50（約225円）
- 月次上限: $30.00（約4,500円）
- 超過時: 全エージェント停止 + LINE通知
- `.env` で調整可能

実際の運用で調整する前提の初期値。
月3万円の予算に対して、上記は保守的な設定（為替150円計算で月約4,500円）。
余裕があれば月次上限を引き上げる。

## セキュリティ設計の考え方

### 脅威モデル

このシステムの主な脅威：

1. **LINE経由の不正指示**: 第三者がWebhookを叩く
   → 署名検証 + User ID許可リスト で排除

2. **サンドボックス脱出**: AIが生成したコードが意図しない操作を行う
   → `/tmp` 隔離 + 30秒タイムアウト + 権限制限

3. **API Key漏洩**: `.env` がリポジトリに入る
   → `.gitignore` で除外。本番環境はSSH鍵認証のみ

4. **コスト暴走**: エージェントが無限ループでAPIを叩く
   → 予算管理で日次/月次上限。超過で全停止

5. **VPSへの不正アクセス**:
   → パスワード認証無効化 + SSH鍵のみ + ufw

### 意図的に対処しないリスク

- DDoS攻撃: 個人利用システムのため優先度低。必要ならCloudflare追加
- データ暗号化: SQLiteのデータは機密度低（ナレッジは本人のみが知る情報）
- マルチユーザー対応: 自分専用なので不要

## 開発上の注意点

### 1. LINE SDKとExpressのbodyパーサー競合

**これは最も頻出するバグの原因。**

LINE Webhookの署名検証は、リクエストのraw bodyに対して行われる。
`express.json()` をグローバルに適用すると、bodyがパース済みになり、
署名検証が失敗する。

```typescript
// NG: express.json()をグローバルに使う
app.use(express.json());
app.use(webhookRouter);  // ← 署名検証が壊れる

// OK: webhookRouter内でline.middleware()が処理
app.use(webhookRouter);  // ← line.middleware()がbodyを処理
// 他のルートには個別にexpress.json()を適用
```

### 2. Claude APIのレスポンスパース

Claude APIは`content`配列を返す。テキストは`type: "text"`のブロックに入る。
ただし、JSONを返すようプロンプトで指示しても、
```json のマークダウンブロックで囲んでくることがある。

taskInterpreter.tsでは、JSONブロックの抽出とフォールバックの両方を実装してある。
新しいエージェントを追加する時も同じパターンを踏襲すること。

### 3. better-sqlite3の同期API

better-sqlite3のメソッド（`.prepare()`, `.run()`, `.get()`, `.all()`）は
全て同期的に実行される。`await` は不要。

ただし、コードの一貫性のためにラッパー関数側ではasync/awaitを使っている
箇所がある（budgetTracker.ts等）。これは将来的にDB層を差し替える可能性を
残すための設計判断。

### 4. サンドボックスの制限

`executor/sandbox.ts` は `child_process.exec` でコードを実行する。
以下の制限がある：

- タイムアウト: 30秒（`config.sandbox.timeoutMs`）
- stdout/stderr: 各5000文字で切り詰め
- 実行後にディレクトリごと削除
- `NODE_ENV=sandbox` が環境変数に設定される

**サンドボックスはセキュリティ境界としては弱い。**
本番環境のファイルシステムにはアクセスできてしまう。
将来的にDockerコンテナ内での実行に移行することを推奨。
ただし初期フェーズでは過剰な複雑さを避けるためこの実装で進める。

### 5. ナレッジDBの検索精度

現在の実装は単純なLIKE検索（部分一致）。
セクション単位で分割してDBに格納し、クエリのキーワードで検索する。

精度が不足する場合の改善パス：
1. 全文検索（SQLiteのFTS5拡張）
2. ベクトル検索（embeddingを生成してコサイン類似度）

ただし、ナレッジファイルは5本・内容は限定的なので、
LIKE検索で実用上の問題が出る可能性は低い。

### 6. エラーログの蓄積

タスクの`error_log`カラムにはJSON配列でエラーが蓄積される。
リトライのたびにエラーメッセージが追加され、
次回のClaude API呼び出し時にコンテキストとして渡される。

これにより「前回と同じアプローチを繰り返す」ことを防ぐ。
3回目以降はOpusに切り替えることで、より高精度な問題分析を行う。

### 7. PM2のログローテーション

PM2のログは放置するとディスクを圧迫する。
デプロイ後に以下を設定すること：

```bash
pm2 install pm2-logrotate
pm2 set pm2-logrotate:max_size 10M
pm2 set pm2-logrotate:retain 7
```

## 将来のエージェント追加時の手順

新しいエージェントを追加する場合：

1. `src/agents/` に新ディレクトリを作成（例: `src/agents/scout/`）
2. `baseAgent.ts` の `Agent` インターフェースを実装
3. システムプロンプトを `prompts.ts` に定義
4. `router.ts` にエージェントを登録
5. `taskInterpreter.ts` のシステムプロンプトに新エージェント名を追加

```typescript
// router.ts への追加例
import { ScoutAgent } from './scout/scoutAgent';
agents['scout'] = new ScoutAgent();
```

タスク解釈エンジンが `agent: "scout"` を返せば、
自動的にScoutAgentにルーティングされる。

## この設計が前提としていること

1. **利用者は1人（Daikiのみ）**: マルチユーザー対応は不要
2. **VPS 1台で完結**: マイクロサービスやk8sは不要
3. **月2〜3万のAPI予算**: コスト効率を常に意識
4. **Claude Codeで開発**: 手書きコードは最小限
5. **段階的リリース**: 完璧を目指さず、動く状態を積み上げる
6. **soicoの事業が最優先**: このシステム開発自体がsoicoの時間を圧迫してはならない
